# Инструкция по работе с Git для публикации изменений

## Простыми словами: что такое Git и зачем он нужен

**Git** — это система контроля версий. Представьте, что это как "история изменений" в Google Docs, но для файлов на вашем компьютере. 

**GitHub** — это облачное хранилище, куда вы загружаете свои файлы, чтобы они были доступны в интернете и не потерялись.

## Процесс публикации изменений (3 простых шага)

### Шаг 1: Проверить, что изменилось
```bash
git status
```
**Что делает:** Показывает список всех файлов, которые вы изменили, добавили или удалили.

**Когда использовать:** Перед тем как что-то закоммитить, чтобы убедиться, что вы не забыли какие-то файлы.

---

### Шаг 2: Подготовить файлы к отправке (добавить в "корзину")
```bash
git add .
```
**Что делает:** Добавляет ВСЕ изменённые файлы в "корзину" (staging area). Точка `.` означает "все файлы в текущей папке".

**Альтернативы:**
- `git add имя_файла.md` — добавить только один конкретный файл
- `git add папка/` — добавить все файлы из конкретной папки

**Когда использовать:** После того как вы внесли изменения в файлы и хотите их сохранить.

---

### Шаг 3: Сохранить изменения с описанием (создать "снимок")
```bash
git commit -m "Краткое описание того, что вы изменили"
```
**Что делает:** Создаёт "снимок" всех изменений с вашим описанием. Это как сохранить версию документа с комментарием "Обновил инструкцию по продажам".

**Примеры хороших описаний:**
- `"docs: добавлена инструкция по работе с CRM"`
- `"fix: исправлена ошибка в регламенте поддержки"`
- `"feat: добавлен новый раздел про маркетинг"`

**Когда использовать:** После `git add`, когда вы готовы сохранить изменения.

---

### Шаг 4: Отправить изменения на GitHub
```bash
git push
```
**Что делает:** Загружает все ваши сохранённые изменения (коммиты) на GitHub, чтобы они были доступны в интернете.

**Когда использовать:** После `git commit`, когда хотите опубликовать изменения на GitHub.

---

## Полный цикл работы (пример)

Допустим, вы обновили файл `ОП/README.md`:

```bash
# 1. Проверяем, что изменилось
git status

# 2. Добавляем изменённый файл в "корзину"
git add ОП/README.md

# 3. Сохраняем изменения с описанием
git commit -m "docs: обновлён README отдела продаж"

# 4. Отправляем на GitHub
git push
```

**Или короче (если изменений много):**
```bash
git add .
git commit -m "docs: обновлена база знаний"
git push
```

## Полезные команды для проверки

### Посмотреть историю коммитов
```bash
git log --oneline -10
```
Показывает последние 10 коммитов с кратким описанием.

### Посмотреть, какие файлы изменены
```bash
git status
```

### Посмотреть разницу в файлах
```bash
git diff
```
Показывает, что именно изменилось в файлах (до того, как вы добавили их через `git add`).

## Частые ситуации

### Ситуация 1: "Я случайно добавил не тот файл"
```bash
# Убрать файл из "корзины" (но изменения останутся)
git restore --staged имя_файла.md

# Или убрать все файлы из "корзины"
git restore --staged .
```

### Ситуация 2: "Я хочу отменить изменения в файле"
```bash
# ВНИМАНИЕ: это удалит все ваши изменения в файле!
git restore имя_файла.md
```

### Ситуация 3: "Я забыл что-то добавить в последний коммит"
```bash
# Добавить файл
git add забытый_файл.md

# Добавить к последнему коммиту (без создания нового)
git commit --amend --no-edit
```

### Ситуация 4: "Git просит username и password при push"
Это нормально! Введите:
- **Username:** `akhromovRT`
- **Password:** ваш Personal Access Token (не пароль GitHub!)

## Рекомендации

1. **Делайте коммиты часто** — лучше много маленьких коммитов, чем один большой
2. **Пишите понятные описания** — через месяц вы сами поймёте, что изменили
3. **Проверяйте `git status` перед коммитом** — убедитесь, что добавляете нужные файлы
4. **Делайте `git push` регулярно** — не накапливайте много коммитов локально

## Быстрая шпаргалка

```bash
# Проверить статус
git status

# Добавить все изменения
git add .

# Сохранить с описанием
git commit -m "описание изменений"

# Отправить на GitHub
git push
```

**Всё! Этих 4 команд достаточно для 99% случаев.**
